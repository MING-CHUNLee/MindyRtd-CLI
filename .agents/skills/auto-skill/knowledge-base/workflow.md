## 🔧 檢查程式碼功能與說明文件連貫性的標準流程
**日期：** 2026-02-26
**情境：** 開發新功能或新指令後，確保所有相關文件與最新的 CLI 工具狀態保持同步。
**最佳實踐：**
- **找出目標文件集合：** 定位專案中的主要文件，例如 `usage.md` (快速參考)、`USER_GUIDE.md` (完整指南)、以及 `README.md`。
- **比對指令與參數：** 檢查程式碼中的每個命令模組 (例如 `src/presentation/commands/` 內的指令) 是否都已更新至快速指南與完整手冊中。
- **驗證範例完整性：** 確保新增的參數（尤其是安全相關功能或實驗性 flag）具備對應的使用情境範例。
- **更新架構紀錄：** 如果新的實作牽涉到系統底層更動 (例如新的溝通機制或權限設計)，須一併更新 `architecture.md` 與對應的設計文件 (`*_DESIGN.md`)。

## 🔧 大規模架構重構的 Commit Message 生成腳本
**日期：** 2026-03-05
**情境：** 當專案發生大規模目錄調整（如從 presentation/commands 遷移至 application/controllers），且涉及 index.ts 註冊邏輯變動時。
**最佳實踐：**
- **追蹤遷移軌跡：** 需明確列出舊路徑與新路徑的對應關係，方便日後追蹤程式碼位置。
- **強調邏輯變更：** 若重構伴隨功能調整（如在入口點新增動態載入 TUI 的邏輯），應單獨列出以區分純重構與功能新增。
- **清理舊文件：** 明確提及已刪除舊目錄中的冗餘文件，確保工作區整潔。
